# Skill: PDF 总结器

读取任意 PDF，提取其文本，产出一篇层次清晰、重点突出、**结论可靠**的中文结构化总结。

## 工作流

### 第 0 步：判断文档体量（决定提取策略）

文档页数直接决定提取方式，**这是最关键的前置判断**：

| 页数 | 类型 | 提取策略 |
|------|------|---------|
| ≤ 5 页 | 短文/单篇 | 直接提取全文，通读后总结 |
| > 5 页 | 长文档（论文/报告/书） | **必须**用 `--locate-summary` 抓结论性章节，见下方铁律 |

判断方法：先运行脚本拿到 `total_pages`（不传 `--max-chars`），看返回的页数再决定。

---

### ⚠️ 铁律一：长文档禁止用 --max-chars 截断

多页论文/报告的**结论性内容永远在文档后部**（各章末尾的"本章小结"、最后的"结论/研究总结"章）。而 `--max-chars` 从头部截断，会**系统性地丢掉所有结论**——结果只剩摘要+目录+绪论，总结时只能凭摘要猜结论，极易出错（真实事故：曾因截断丢失结论章，把"0° 最优"错写成"负向偏置最优"，方向相反）。

- **短文档（≤5 页）**：可以不用截断，全文通常不会撑爆上下文。
- **长文档（>5 页）**：**不要**传 `--max-chars`。全文较大时，靠 `--locate-summary`（铁律二）定位结论段，只把结论段和摘要读进上下文即可，正文细节按需在脚本输出里切片。

### ⚠️ 铁律二：结论性陈述必须以正文小结原文为准

**摘要 ≠ 结论的可靠来源。** 摘要为求凝练，常与正文细节甚至方向性结论不符。生成总结时：

1. 先读**摘要** → 建立背景、研究问题、方法、总体方向的认知（这些摘要通常可靠）。
2. 凡是要写进总结的**具体结论**（最优参数、数据、排序、因果关系），**必须**回到正文各章"本章小结"和末尾"结论/研究总结"章的原文核对，逐条对齐。
3. 摘要里读到的结论方向，若与正文小结冲突，**以正文小结为准**，并在总结中采用正文的数据表述。

这两条铁律是本 skill 区别于"粗暴提取+凭印象总结"的核心，违反则结论不可信。

---

### 第 1 步：验证文件存在

调用前先用 `dir`（Windows）或 `ls`（Unix）确认路径有效、文件非空。路径含空格或中文时务必用双引号包裹。

### 第 2 步：提取文本

**始终使用本 skill 自带的脚本，不要自己写 Python 或直接调 pdfplumber**——脚本已封装好跨平台解释器适配、依赖自动安装、扫描件识别、结论段定位：

```bash
python "<skill_dir>/scripts/extract_pdf.py" "<pdf_path>"
```

- 用 `python`，**不要**用 `python3`（Windows 默认没有该别名，会报 9009）。
- `<skill_dir>` 即本 SKILL.md 所在目录。
- 脚本首次运行会自动 `pip install --user pdfplumber`，无需手动装依赖。

**输出 JSON 字段**：

| 字段 | 含义 |
|------|------|
| `status` | `success` / `scanned` / `error`，决定后续处理 |
| `total_pages` | 总页数，用于第 0 步体量判断 |
| `total_chars` | 总字符数 |
| `text` | 全文（不要截断长文档） |
| `per_page` | 每页字符数，可用于发现异常（如某页 0 字符=图片页） |
| `summary_segments` | 仅当传 `--locate-summary` 时存在；自动定位的结论性章节段 |

**status 处理**：

| status | 含义 | 处理 |
|--------|------|------|
| `success` | 提取成功 | 进入第 3 步 |
| `scanned` | 疑似扫描件（平均每页 <50 字） | 按返回的 `note` 建议 OCR（ocrmypdf/Tesseract）或渲染成图片用 Read 工具读取。向用户说明并征询是否继续 |
| `error` | 提取失败（损坏/加密/无权限） | 把 `note` 里的原因告诉用户，不要编造内容 |

**长文档专用提取命令**（>5 页）：

```bash
python "<skill_dir>/scripts/extract_pdf.py" "<pdf_path>" --locate-summary --segment-chars 800
```

- `--locate-summary`：扫描全文，定位"本章小结/结论/研究总结/展望/Summary/Conclusion"等结论性章节，**自动过滤目录项**（关键词后紧跟连续点号+页码的行是目录，不是正文）。
- `--segment-chars 800`：每个结论段取关键词后 800 字符（默认 700）。论文小结通常 200-400 字，800 字够覆盖一个小结全文。
- 返回的 `summary_segments` 是 `[{keyword, pos, text}]` 数组，**把这些 text 通读一遍**——它们就是全文所有结论的原文集合。

### 第 3 步：通读并生成结构化总结

对长文档：把**摘要**（建立背景/方向）+ **summary_segments**（核对具体结论）一起读进上下文，然后按格式规范输出。短文档则直接通读全文。

**严禁照搬原文段落**——总结的价值在于提炼与重组，不是复制粘贴。

## 总结格式规范

默认用中文输出（用户用英文提问则用英文）。按下列层次组织，可依内容长短增删模块，但**核心主题**和**一句话主旨**两块必须有：

### 1. 标题行
`# 📄 文档总结：《文章标题》`——标题从正文推断，不要用文件名。

### 2. 核心主题（2-4 句）
说清这是什么类型的文档、解决/讨论什么问题、面向谁。让没看过原文的人 30 秒内建立认知。

### 3. 正文分点（文档的核心骨架）
- 若文档本身有清晰的分点/章节结构（如"三个阶梯""五步法"），**沿用其结构**，用 `###` 三级标题逐个展开。
- 每个要点下用**简短描述 + 表格/列表/引用块**组合呈现，避免大段文字。表格适合对比型内容（参数对比、阶段对比），引用块适合点睛金句或数据。
- 用 emoji 作视觉锚点要克制，每个要点开头一个即可，不要每行都加。

### 4. 一句话主旨（必须有）
用 `>` 引用块给出全文最凝练的概括。这是整篇总结的"电梯演讲"。

### 5. 后续建议（可选）
列 2-3 个可延伸的动作，例如：导出为文档、做成思维导图、应用到具体场景。让用户感到"还能继续往前走"。

---

## 写作要点

- **提炼而非复述**：把原文 5 句压成 1 句，把散落的要点收拢成表格。读者要的是地图，不是路标清单。
- **保留关键术语与数据**：专业概念、具体数值、最优参数、方法名要原样保留，不要为了通俗而失真。尤其结论里的数字（如"4.5mm 最优""OTDF=0.21"）必须精确。
- **诚实标注不确定性**：若 PDF 提取质量差（乱码、缺页、OCR 噪声），或某部分未完整读取，在总结里说明，不要假装内容很完整。
- **长度自适应**：1 页短文总结控制在 300-500 字；多页长文档可分章节详述，但每节仍要精炼。

---

## 交付

总结完成后：
1. 用中文呈现完整总结（上述格式）。
2. 报告文件元信息：总页数、总字符数（从 JSON 的 `total_pages`、`total_chars` 取）。
3. 主动询问是否需要导出（Word/PDF）或做可视化（思维导图/信息图）。

不要把提取脚本的原始 JSON、调试日志、中间过程暴露给用户——他们要的是总结，不是流水账。
